/INTERNAL PROCESSES

How we operate

These processes keep us organized and make sure nothing falls through the cracks.

Core PM processes

Meeting notes protocol ▶
After every client meeting, create a dated note in 06-Clients/[name]/meeting-notes/ (use the meeting notes template). Include: attendees, agenda, decisions made, action items, and signals extracted. Update the client's _overview.md with any status changes. Log all recommendations in recommendations-tracker.md.
Client health scoring ▶
Every active client gets a health score (0-100) based on 6 dimensions: Data Quality (20%), Engagement Frequency (20%), Recommendation Adoption (20%), Revenue/Performance Trend (20%), Contract Health (10%), Vault Completeness (10%). Score thresholds: 85-100 Healthy, 70-84 Watch, 50-69 At Risk, 0-49 Critical (immediate escalation to Walter). Update health scores quarterly or after significant changes.
Signal extraction from communications ▶
When processing any client communication (email, call, meeting, update email), extract qualitative signals and append to the client's _overview.md under ## Client Signals. Categories: FR (Feature Request), OB (Objection), SAT+/SAT- (Satisfaction), PS (Priority Shift), EX (Expansion), CH (Churn), DA (Data/Access), IC (Industry Context). Format: - [YYYY-MM-DD] [CODE] — [description] (source: email/meeting)
Recommendation tracking & learning loop ▶
Every recommendation you make to a client gets logged in their recommendations-tracker.md. Track: what was recommended, when, linked pattern (if applicable), priority (High/Medium/Low), whether it was actioned, and the outcome. This feeds the cross-client learning system — it's how we learn which advice actually works. Close the loop: measure if the recommendation moved the needle.
Escalation paths ▶
Health score 0-49 (Critical): Same-day alert to Walter. Payment issues: Flag in Attio + notify Walter within 24h. Technical emergency (tracking completely down): Escalate to Walter immediately. Client mentions leaving: CH signal + direct escalation to Walter same day. Scope creep: Document in meeting notes, discuss with Walter before agreeing to additional work. Data access problems: Flag immediately, block reporting if needed, escalate if client can't/won't fix.
Monthly reporting workflow ▶
Monthly reports due by the 5th of each month. Workflow: (1) Pull data from GA4, run analysis. (2) Check for anomalies vs. prior month. (3) Generate performance report using the monthly report template. (4) Include: traffic trends, conversion metrics, consent impact analysis, any anomalies detected, and recommendations. (5) Add client commentary & next month's priorities. (6) Send to client + log report in vault under 06-Clients/[name]/. (7) Schedule follow-up call to review.

Hands-on GA4 & GTM proficiency

As a PM, you need to QA tracking, pull reports, and troubleshoot basic issues. Here's what you need to know.

Navigating GA4

  • Real-time reports: Verify events firing live
  • Acquisition reports: Traffic sources, campaign performance
  • Engagement reports: Events, conversions, pages, sessions
  • Monetization reports: E-commerce, revenue, ROAS
  • Explore: Custom funnels, path analysis, cohort analysis

Key metrics you'll reference: Sessions, Users, Conversion Rate, Revenue, ROAS, Avg Session Duration, Bounce Rate.

GA4 QA Checklist

  • Events tab shows expected events (page_view, purchase, etc.)
  • Conversions are marked correctly (not just events)
  • User properties configured (subscription_status, lifetime_value, etc.)
  • Data streams active & receiving data
  • Enhanced measurement enabled (scroll, video, search, site search)
  • Google Signals enabled (for cross-device)
  • Data retention set to 14 months
  • Property timezone matches client location

Reading a GTM Container

  • Tags: What fires (GA4 config tag, conversion linker, ad platform pixels, etc.)
  • Triggers: When they fire (page view, click, form submission, scroll, custom event)
  • Variables: Data they use (page URL, element text, GTM data layer values)
  • Container versions: Each version is a snapshot. Check the latest is live.

You don't need to build tags, but you need to read and QA them. Use Preview mode to verify.

GTM Preview Mode (Essential)

  • Enter preview mode: Click "Preview" in GTM. Copy unique URL to browser.
  • See what fires: "Tags Fired on This Page" shows which tags + triggers executed.
  • Check data layer: Data layer shows all variables pushed by the website.
  • Spot errors: Red = tag blocked/failed. Yellow = tag defined but not fired. Green = success.
  • Test consent: Reject consent, refresh page, see if GA4 tag fires (it shouldn't).

Why Server-Side GTM

Client-side GTM: Data sent from the browser → ad blockers, consent tools, privacy extensions can block it. Result: 20-40% lost conversions.

Server-side GTM: Data sent from our server (Stape) → recovers cookieless data, bypasses blockers, respects consent. We recover 20-40% of lost conversions for every client.

Bottom line: Server-side is a key value driver. Clients expect it at T2/T3 tiers.

What to Check

  • Server container URL active & DNS resolving
  • Client-side to server-side forwarding working (check GTM preview)
  • Enhanced conversions configured (match email, phone, address)
  • First-party cookies set correctly (check browser Network tab)
  • Consent signals passed through (consent_mode parameters)
  • Event payloads complete (not truncated)
  • No 500 errors in Stape logs

Consent Mode v2

What it is: Google's framework for adjusting tag behavior based on user consent signals. Two modes: Basic (block tags without consent) and Advanced (send cookieless pings for modeling).

What we do: Implement Advanced mode for most clients. Result: recovers measurement data Google can model even without full consent. 34% of sessions are hidden by consent — this is the gap we solve.

What to Verify

  • CMP banner loads on first visit (Cookiebot, Usercentrics, etc.)
  • Consent choices stored correctly (localStorage/cookies)
  • GA4 consent_state parameters firing correctly
  • Ad platform tags respect consent (no firing without user consent)
  • Server-side consent propagation working
  • 34% sessions hidden by consent — this is the measurement gap we solve
