/01 · DEFINITION
/WHAT IS AN EVALUATION HARNESS

A framework that runs a model against a fixed task suite and scores the outputs.

01

Task suite

Questions + expected answers or rubric.

02

Runner

Executes the system under test.

03

Grader

Deterministic checks or LLM-as-judge.

04

Aggregator

Combines runs, computes variance.

05

Reporter

Surfaces results, trends, regressions.

Examples: Inspect · promptfoo · lm-evaluation-harness · HELM

/05 · TASK CATEGORIES

Five categories — so failures tell us where, not just whether.

FACTUAL

T-01, T-02, T-06

Look up a specific metric. Deterministic grading, exact match.

COMPUTATION

T-03, T-04

Derive values from multiple inputs — ratios, run-rates, deltas.

ATTRIBUTION

T-05, T-09

Map values across channels or time windows. Named-entity exact match.

REASONING

T-07

Explain why something is happening. LLM-as-judge.

JUDGMENT

T-08, T-10

Prioritise. Recommend. Rank. LLM-as-judge + actionability score.

A prompt that aces factual but tanks judgment is a different fix than one that's shaky on computation. Categorising isolates the failure mode.

/06 · RUBRIC

Grade what it produced, not the path it took.

/DETERMINISTIC TASKS

Numeric tolerance bands

  • Pass — values in tolerance, entities exact.
  • Partial — 2× tolerance, or one entity fuzzy but recognisable.
  • Fail — anything worse.
  • Fabrication → auto-Fail. Always. Even if everything else is right.
/LLM-AS-JUDGE TASKS

Four-criteria scoring (0–2 each)

  • Factual grounding — no fabrication
  • Reasoning quality — internally consistent
  • Completeness — addresses all parts
  • Actionability — concrete recommendation (T-08, T-10)

≥75% total → Pass · 50–74% → Partial · <50% → Fail

Reward-hacking guardrail: The judge never sees expected answers. The model under test never sees the rubric. Anti-gaming by construction — see [[reward-hacking-in-autoresearch-loops]].
