The concept

Email is the input. The vault is the memory.

Most update work happens in email — a client says yes, a contractor sends a doc, a prospect asks a question. Instead of copy-pasting into Notion, we label. Claude reads labelled emails, extracts the signal, and writes to the right place in the vault.

update_odisei: Gaby confirmed Q2 dashboard scope
From: walter@ampliodata.ioTo: hello@ampliodata.io
Amplio Updates
Quick one so Claude picks it up — Gaby signed off on the Q2 dashboard scope. Adding a repeat_purchase_cohort metric, removing the old store-level split. Starting implementation Monday. Update 06-Clients/Odisei/_overview.md — status stays Active, note the scope change, add a recommendation row.
The flow

Five steps, always the same.

Every update_ email runs through the same pipeline. Deterministic, auditable, reversible.

/ Step 01

Send / label

Write an email with update_ prefix, or apply the Amplio Updates label.

/ Step 02

Gmail MCP fetch

Claude calls gmail_search_messages for label:"Amplio Updates" is:unread.

/ Step 03

Parse & route

Subject prefix identifies the client / area. Body extracts the signal.

/ Step 04

Vault write

Edits land in _overview.md, tracker, pipeline, or Galaxy note with provenance.

/ Step 05

Close the loop

Claude replies with a diff summary + Attio sync status. Loop closed.

Triggers & commands

Subject-line grammar.

The subject line is a mini-command. Prefixes route the email to the right corner of the vault.

update_[client]:Updates that client's _overview.md. Adds a signal. Logs recommendation if one is present.
newclient_[name]:Creates 06-Clients/[name]/ from the template. Updates Client-Index. Seeds data-context.
newprospect_[name]:Creates prospect in _Prospects/. Updates Pipeline Dashboard. Scores lead if ICP data attached.
meeting_[client]:Creates dated file in [client]/meeting-notes/. Extracts signals. Logs recommendations to tracker.
graduate_[name]:Promotes prospect → client. Moves folder, expands template, archives pipeline entry.
update_amplio:Internal. Updates 02-Areas/Amplio-Internal/. Never routed to 06-Clients/.

Safety rules

The six rules this system obeys.

01

No silent writes

Every edit is summarised back to the sender in a reply. If you didn't see a reply, nothing happened.

02

Label scoped to @ampliodata.io

Only emails from a verified Amplio address are trusted for vault writes. Client-forwarded emails must be manually labelled by Walter.

03

Never delete

Claude edits and appends. Never deletes vault content from an update_ email. Removal requires an explicit confirmation step.

04

Provenance everywhere

Every auto-edit gets source: email/[gmail-thread-id] + timestamp in the edit commit or inline comment.

05

Ambiguity stops the loop

If routing is unclear, Claude replies asking for clarification instead of guessing. Under-writing beats over-writing.

06

Label gets removed on success

Processed emails lose the "Amplio Updates" label. The inbox becomes a natural work queue. If it's labelled, it's pending.
